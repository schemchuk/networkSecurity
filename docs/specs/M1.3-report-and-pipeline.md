# M1.3 — Report renderer + наскрізний orchestrator

**Контекст:** фінальний блок M1 — замикає MVP-вертикаль (§9 ARCHITECTURE.md):
`nmap.xml → parser (M1.1) → ReconAgent (M1.2) → report.md`. Після цього MVP готовий.

## Ціль
- `tools/report_render.py` — рендер `list[Finding]` → Markdown-звіт.
- `agents/pipeline.py` — функція наскрізного прогону (тестована, з ін'єкцією LLM).
- `scripts/run.py` — CLI-обгортка, що запускає весь конвеєр над реальним Ollama.

## Файли

### `tools/report_render.py`
- `def render_report(findings: list[Finding], manifest: Manifest | None = None) -> str`:
  Markdown із секціями:
  - Заголовок `# Recon Report — <run_id|"—">`; якщо є manifest — рядки scenario/targets/
    operator/started.
  - `## Summary` — таблиця: `ID | Host | Port | Service | Product | Version | Severity`.
  - `## Findings` — по кожному finding блок:
    `### <id> — <host>:<port>/<protocol> <service>`,
    `**Summary:**` (або `_(no summary)_` якщо порожньо),
    `**Next steps:**` списком (пропустити, якщо порожньо),
    якщо є `cves` — рядок з їх переліком (на майбутнє; зараз зазвичай порожньо).
  - Порожній список findings → секція з `_No open services found._`.
- `def write_report(findings, path: str | Path, manifest=None) -> None` — записати у файл (utf-8).
- Без LLM/мережі.

### `agents/pipeline.py`
- `def run_recon_pipeline(nmap_path, run_id, runs_dir="runs", llm=None) -> Path`:
  1. `findings = parse_nmap_xml(nmap_path)` (M1.1);
  2. `agent = ReconAgent(run_id, runs_dir=runs_dir, llm=llm)`; `findings = agent.run(findings)`;
  3. відрендерити звіт, записати в `runs_dir/<run_id>/report.md`;
  4. повернути `Path` до report.md.
- Якщо `llm` передано — використати його (для тестів); інакше ReconAgent створить реальний.

### `scripts/run.py` — CLI (executable)
- Аргументи: `--nmap` (обов'язк.), `--run-id` (default: авто `YYYY-MM-DD_HHMMSS_<scenario>`),
  `--scenario` (default `recon`), `--runs-dir` (default `runs`),
  `--targets`, `--operator` (default `$USER`).
- Створити run-директорію та `manifest.json` (через `Manifest`, як у `new_run.py`;
  можна винести спільний хелпер або продублювати мінімально — на твій розсуд, познач у звіті).
- Викликати `run_recon_pipeline(...)` з реальним `LLMClient()`.
- Надрукувати шлях до `report.md`. Docstring: потрібен запущений `ollama serve`.

## Тести — `tests/test_report_and_pipeline.py` (мок LLM, без мережі)
- `test_render_report_contains_findings`: рендер 2 findings → рядок містить host, service,
  обидва id, заголовки `## Summary` і `## Findings`.
- `test_render_empty`: порожній список → містить `_No open services found._`.
- `test_pipeline_end_to_end`: з фейковим LLM (як у M1.2) + фікстурою
  `tests/fixtures/nmap_sample.xml`, `runs_dir=tmp_path`:
  - повертає шлях до `report.md`, файл існує;
  - `runs/<run_id>/findings.json` існує і парситься у `list[Finding]` (2 шт.);
  - у звіті присутні `microsoft-ds` і `10.10.10.5`.
