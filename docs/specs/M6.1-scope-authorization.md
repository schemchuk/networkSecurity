# M6.1 — Scope / авторизація (легальна основа реального прогону)

**Контекст:** старт M6 (див. [ROADMAP-M6-professional.md](../ROADMAP-M6-professional.md)).
Перед запуском проти реальної системи інструмент має **вимагати дозвіл і перевіряти межі**.
Ціль поза дозволеним списком → відмова працювати. Це і легальна вимога, і захист від
випадкового виходу за scope.

## Ціль
Модуль `tools/scope.py`, що описує engagement (дозвіл + дозволені цілі) і перевіряє, чи ціль
у межах. Інтеграція в `scripts/run.py`: якщо передано engagement — усі цілі та хости зі скану
мають бути in-scope, інакше запуск блокується.

## Файли

### `tools/scope.py`
- `class ScopeError(Exception)`.
- `class Engagement(BaseModel)`:
  - `client: str` — назва замовника;
  - `authorized_by: str` — хто дав дозвіл;
  - `authorization_ref: str` — посилання на документ/тікет дозволу;
  - `authorized_targets: list[str]` — дозволені цілі: окремі IP, CIDR (напр. `192.168.56.0/24`)
    або hostname;
  - `start_date: date | None = None`, `end_date: date | None = None` — вікно робіт (опційно).
- `def load_engagement(path: str | Path) -> Engagement` — читає YAML, валідує.
- `def is_in_scope(target: str, engagement: Engagement) -> bool`:
  - для кожного запису в `authorized_targets`:
    - якщо запис — валідний CIDR → перевірити, чи `target` (IP) у цій мережі
      (модуль `ipaddress`);
    - якщо запис — валідний IP → точний збіг;
    - інакше (hostname) → точний збіг рядків (case-insensitive).
  - `target`, що не парситься як IP, порівнюється лише з hostname-записами.
  - повертає `True`, якщо хоч один запис підходить.
- `def assert_in_scope(target: str, engagement: Engagement) -> None` — якщо не in-scope,
  підняти `ScopeError` з чітким повідомленням (яка ціль, який engagement).
- (опційно) `def check_window(engagement, when: date) -> bool` — чи дата в межах вікна;
  якщо дат нема — завжди `True`. Використання — на розсуд; познач у звіті Kimi.

### `configs/engagement.example.yaml` — шаблон
```yaml
client: "Example Corp"
authorized_by: "Jane Doe, CISO"
authorization_ref: "PENTEST-2026-001 (signed 2026-08-01)"
authorized_targets:
  - "192.168.56.0/24"
  - "10.0.0.5"
  - "app.example-lab.local"
start_date: "2026-08-01"
end_date: "2026-08-15"
```

### `scripts/run.py` — інтеграція
- Додати аргумент `--engagement <path>` (опційний).
- Якщо переданий:
  - завантажити engagement;
  - для кожної цілі з `--targets` викликати `assert_in_scope` — при `ScopeError`
    вивести зрозумілу помилку в stderr і **вийти з кодом 2** (не запускати pipeline);
  - після парсингу nmap — перевірити, що кожен хост зі знахідок in-scope; хост поза scope →
    така сама відмова (захист, якщо у XML потрапило зайве).
- Якщо `--engagement` **не** переданий — вивести помітне попередження в stderr:
  `WARNING: no engagement scope provided — LAB USE ONLY. Do not run against systems you are
  not authorized to test.` — і продовжити (сумісність з наявним lab-запуском).

## Тести — `tests/test_scope.py` (без мережі)
- `test_exact_ip_in_scope`: `10.0.0.5` в межах, `10.0.0.6` — ні.
- `test_cidr_in_scope`: `192.168.56.101` в межах `192.168.56.0/24`; `192.168.57.1` — ні.
- `test_hostname_in_scope`: точний збіг hostname (case-insensitive); інший — ні.
- `test_assert_raises_out_of_scope`: `assert_in_scope` кидає `ScopeError` для чужої цілі.
- `test_load_engagement`: читає `configs/engagement.example.yaml` у валідний `Engagement`.
