# M3.2 — Hardening у pipeline + mitigation/detection у звіті

**Контекст:** замикає M3. Додає HardeningAgent (M3.1) у наскрізний прогін і показує
mitigation/detection у звіті. Повний потік:
`nmap → parse → Recon → VulnAnalysis → Hardening → report.md`.

## Файли

### `agents/pipeline.py` — додати крок hardening
- Імпортувати `HardeningAgent`.
- Після `VulnAnalysisAgent(...).run(...)` додати:
  `HardeningAgent(run_id, runs_dir=runs_dir, llm=llm).run(findings)`.
- Порядок важливий: hardening відбирає findings за CVE/severity, які проставляє
  vuln-агент, тому має йти після нього. Далі — рендер звіту як було.

### `tools/report_render.py` — показати mitigation/detection
- У блоці кожного finding (`## Findings`), після `**CVEs:**` (якщо є), додати:
  - `**Mitigation:**` — якщо `finding.mitigation` непорожній;
  - `**Detection:**` — якщо `finding.detection` непорожній.
- Порожні поля не виводити (щоб findings без blue-team порад не смітили).
- Решту (Summary, сортування за priority, екранування) не чіпати.

## Тести — оновити `tests/test_report_and_pipeline.py`
- **Оновити фейковий LLM** у `test_pipeline_end_to_end`, щоб повертав правильний тип
  залежно від `schema`:
  ```python
  def chat_json(self, prompt, schema, system=None, retries=1):
      if schema is HardeningAdvice:
          return HardeningAdvice(mitigation="Upgrade the service.",
                                 detection="Watch auth logs.")
      return ReconEnrichment(summary="...", next_steps=["..."])
  ```
  (імпортувати `HardeningAdvice` з `agents.hardening`).
- Додати перевірки: у `report.md` для finding зі "Samba" (є CVE) присутні
  `**Mitigation:**` і `**Detection:**` з відповідним текстом.
- **Додати** `test_render_shows_mitigation_detection`: finding із заповненими
  `mitigation`/`detection` → рядки присутні у звіті; finding з порожніми — рядків нема.
- Наявні render-тести лишити зеленими.
