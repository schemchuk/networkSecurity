# M2.2 — Vuln Analysis агент (детермінований, без LLM)

**Контекст:** M2, блок 2. Інтегрує `search_cves` (M2.1) у findings і пріоритезує.
**Свідоме рішення:** агент **повністю детермінований, без LLM-викликів** — тому швидкий
і стабільний на слабкому залізі (див. пам'ять про hardware-constraint). CVE — з реального
exploitdb, не від моделі (§5.4 ARCHITECTURE.md).

## Ціль
`agents/vuln_analysis.py`: для кожного finding дістати реальні CVE, проставити severity
та priority за прозорою евристикою, перезаписати `findings.json`.

## Файли

### `agents/base.py` — маленька зміна: лінива LLM
Зараз `__init__` завжди створює `LLMClient()`, якщо `llm is None`. Детермінованому агенту
клієнт не потрібен. Зробити ліниво:
- зберігати `self._llm = llm`;
- додати `@property def llm(self)`: якщо `self._llm is None` — створити `LLMClient()` при
  першому доступі й закешувати; інакше повернути наявний.
- Поведінка для ReconAgent не змінюється (він звертається до `self.llm`). Переконатись,
  що наявні тести recon лишаються зелені.

### `agents/vuln_analysis.py`
- `from tools.cve_lookup import search_cves, CVELookupError`.
- `class VulnAnalysisAgent(BaseAgent)`, `name = "vuln_analysis"`:
  - `__init__(self, run_id, runs_dir="runs", llm=None, cve_lookup=search_cves)`:
    виклик `super().__init__(...)`; зберегти `self.cve_lookup` (ін'єктується в тестах).
  - `run(self, findings: list[Finding]) -> list[Finding]`:
    - emit `note` старту (data: count).
    - `lookup_disabled = False`.
    - для кожного finding:
      - якщо `lookup_disabled` або немає `product` → пропустити lookup;
      - інакше: `try: new = self.cve_lookup(finding.product, finding.version)`
        - об'єднати з `finding.cves` без дублів (за `id`);
        - emit `note` (ref=finding.id, data: {"cves": <к-сть нових>});
      - `except CVELookupError:` emit `error` (ref=finding.id, data: причина),
        виставити `lookup_disabled = True` (бінарник відсутній — далі не пробувати).
    - **severity (евристика):** finding з ≥1 CVE → `Severity.HIGH`
      (наявність публічного експлойта в exploitdb — сильний сигнал); інакше не чіпати.
    - **priority:** ранжувати всі findings за ключем `(-len(cves), port or 0)`;
      призначити `priority = 1..N` (1 = найвищий). Порядок самого списку не міняти,
      лише проставити поле `priority`.
    - перезаписати `self.run_dir/"findings.json"` (JSON-масив, indent=2).
    - повернути список.

## Тести — `tests/test_vuln_analysis_agent.py` (ін'єкція cve_lookup, без searchsploit)
- Фейковий `cve_lookup`, що повертає CVE для певного product (напр. Samba → `CVE-2007-2447`),
  інакше `[]`.
- `test_attaches_cves`: finding із Samba отримує CVE; finding без — лишається без.
- `test_sets_high_severity_for_cve`: finding із CVE → `severity == HIGH`; без CVE → не HIGH.
- `test_priority_orders_cve_first`: finding із CVE має менший (кращий) `priority`, ніж без.
- `test_dedupes_existing_cve`: якщо finding вже має цей CVE — дубля не з'являється.
- `test_tolerates_missing_searchsploit`: `cve_lookup`, що кидає `CVELookupError` →
  агент не падає, пише `error`-подію, findings лишаються без CVE.
- `test_writes_findings_json`: `findings.json` існує і парситься у `list[Finding]`.
- `runs_dir=tmp_path`; llm не потрібен (передавати `llm=object()` або лишити default —
  завдяки лінивій властивості клієнт не створюється).
