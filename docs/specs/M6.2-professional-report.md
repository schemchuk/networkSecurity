# M6.2 — Професійний звіт (executive summary, методологія, CVSS, ризик, рекомендації)

**Контекст:** продовження [ROADMAP-M6-professional.md](../ROADMAP-M6-professional.md) після
[M6.1 (scope/авторизація)](M6.1-scope-authorization.md), уже змерджено в `main`. Зараз звіт
(`tools/report_render.py`) — це таблиця + список знахідок, без резюме для замовника. Мета M6.2:
додати шар "для менеджера/замовника" зверху наявних технічних деталей, не ламаючи їх.

## Ціль
`render_report` / `write_report` отримують новий необов'язковий параметр `engagement` і
генерують три нові секції перед наявною `## Summary`: **Engagement**, **Executive Summary**,
**Methodology**. Таблиця `## Summary` і кожна знахідка в `## Findings` отримують колонку/рядок
**CVSS**.

## Файли

### `tools/report_render.py` — зміни
Сигнатури (тип `Engagement` — з `tools.scope`, імпорт лише під `TYPE_CHECKING`, як вже зроблено
для `Finding`/`Manifest`):

```python
def render_report(
    findings: list["Finding"],
    manifest: "Manifest" | None = None,
    engagement: "Engagement" | None = None,
) -> str: ...

def write_report(
    findings: list["Finding"],
    path: str | Path,
    manifest: "Manifest" | None = None,
    engagement: "Engagement" | None = None,
) -> None: ...
```

Порядок секцій у виводі (усі — після заголовка `# Recon Report — {run_id}`):

1. **`## Run metadata`** — як зараз, без змін.
2. **`## Engagement`** — лише якщо переданий `engagement`:
   - `- **Client:** {engagement.client}`
   - `- **Authorized by:** {engagement.authorized_by}`
   - `- **Authorization ref:** {engagement.authorization_ref}`
   - `- **Window:** {start_date} – {end_date}` (якщо обидві дати є); якщо якоїсь нема —
     `"open-ended"` замість неї; якщо обох нема — рядок `- **Window:** open-ended`.
3. **`## Executive Summary`** — завжди:
   - `- **Total findings:** N`
   - `- **By severity:** critical: X, high: Y, medium: Z, low: W, info: V` (виводити лише
     ненульові severity, у порядку critical→high→medium→low→info; якщо все 0 — `none`)
   - `- **Overall risk:** {rating}` — найвища severity серед знахідок, Title Case
     (`Critical`/`High`/`Medium`/`Low`/`Info`); якщо знахідок нема — `None`.
   - `**Top recommendations:**` — нумерований список (макс. 5) з `finding.mitigation` для
     знахідок з severity `critical` або `high`, у порядку `priority` (як `_priority_key`),
     без дублікатів (порівняння точного рядка), пропускаючи порожні `mitigation`. Якщо після
     фільтрації список порожній — рядок `_No high-priority mitigations identified._`.
4. **`## Methodology`** — фіксований статичний текст (без даних рану), коротко описує етапи:
   nmap service/version scan → парсинг у `Finding` → Recon-агент (LLM: summary, next steps) →
   Vuln Analysis-агент (кореляція CVE, severity, priority) → Hardening-агент (mitigation,
   detection). ~4-6 рядків, без зайвого маркетингу.
5. **`## Summary`** — наявна таблиця, додати колонку **CVSS** останньою:
   `| ID | Priority | Host | Port | Service | Product | Version | Severity | CVSS |`.
   Значення — максимальний `cve.cvss` серед `finding.cves` (формат `f"{value:.1f}"`), або `""`
   якщо CVE немає або жоден не має `cvss`.
6. **`## Findings`** — наявний рендер; у рядку `**CVEs:** ...` формат кожного CVE:
   `{cve.id}` якщо `cve.cvss is None`, інакше `{cve.id} (CVSS {cve.cvss:.1f})`.

Нові приватні хелпери (аналогічно наявним `_escape_pipe`, `_priority_key`):
- `_max_cvss(finding) -> float | None`
- `_overall_risk(findings) -> str`
- `_top_recommendations(findings, limit: int = 5) -> list[str]`
- `_format_cve(cve) -> str`

### `agents/pipeline.py` — зміни
`run_recon_pipeline` отримує новий необов'язковий параметр `engagement: "Engagement" | None =
None` (import під `TYPE_CHECKING` з `tools.scope`) і просто прокидає його у виклик
`write_report(enriched, report_path, manifest=manifest, engagement=engagement)`.

### `scripts/run.py` — зміни
Якщо `args.engagement` передано — після успішної перевірки scope передати вже завантажений
`engagement` в `run_recon_pipeline(..., engagement=engagement)`. Якщо не передано — не передавати
(лишається `None`, звіт без секції Engagement).

## Тести — доповнити `tests/test_report_and_pipeline.py`
- `test_render_report_engagement_section`: з `engagement` — секція `## Engagement` присутня і
  містить client/authorized_by/authorization_ref; без `engagement` — секції нема в тексті.
- `test_render_report_executive_summary_counts`: findings з різними severity → правильні
  лічильники та `Overall risk` = найвища severity.
- `test_render_report_top_recommendations_dedup_and_limit`: >5 унікальних mitigation серед
  critical/high → рівно 5 у виводі, дублікати не повторюються.
- `test_render_report_no_high_priority_mitigations`: усі findings severity info/low →
  `_No high-priority mitigations identified._`.
- `test_render_report_cvss_column_and_cve_format`: finding з `CVE(id=..., cvss=7.5, source=...)`
  → у таблиці `7.5`, у деталях `CVE-XXXX-YYYY (CVSS 7.5)`; finding без cvss → `""` у таблиці,
  голий `CVE-...` у деталях.
- Існуючі тести (`test_render_report_contains_findings` та інші) не повинні зламатися — секції
  Engagement/Executive Summary/Methodology не заважають наявним asserts (вони шукають підрядки,
  не порівнюють весь текст).
