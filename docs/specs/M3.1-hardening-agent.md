# M3.1 — Hardening/Detection advisor агент

**Контекст:** старт M3. Blue-team перспектива (§5.2 ARCHITECTURE.md): для кожного
пріоритезованого finding — як закрити (mitigation) і що бачить жертва (detection).
Дотримуватись `docs/SCOPE.md`: оборонний фокус, жодного офенсивного коду.

**Рішення під слабке залізо:** агент робить LLM-виклик **лише для пріоритезованих
findings** (де є CVE або severity high/critical), а не для всіх портів — щоб не витрачати
повільний інференс намарно. Спирається на severity/priority з M2.2.

## Файли

### `agents/hardening.py`
- `class HardeningAdvice(BaseModel)`: `mitigation: str`, `detection: str`.
- `HARDENING_SYSTEM_PROMPT: str` — роль blue-team аналітика: для заданого сервісу/CVE
  дати (1) mitigation — як усунути/пом'якшити, (2) detection — які логи/алерти/артефакти
  побачить захисник. Тільки оборона; без експлойтів/пейлоадів. (2–5 речень.)
- `class HardeningAgent(BaseAgent)`, `name = "hardening"`:
  - `def _should_enrich(self, finding: Finding) -> bool`:
    `True`, якщо `finding.cves` непорожній **або** `finding.severity in {HIGH, CRITICAL}`.
  - `run(self, findings: list[Finding]) -> list[Finding]`:
    - emit `note` старту (data: {"total": len, "selected": <к-сть, що проходять _should_enrich>}).
    - для кожного finding, що проходить `_should_enrich`:
      - побудувати короткий промпт (host/port/service/product/version + список CVE id);
      - `self.llm.chat_json(prompt, schema=HardeningAdvice, system=HARDENING_SYSTEM_PROMPT)`;
      - записати `finding.mitigation`, `finding.detection`;
      - emit `llm_call` (ref=finding.id);
      - `except LLMError:` emit `error` (ref=finding.id), не падати.
    - findings, що не пройшли відбір, лишаються без mitigation/detection (порожні).
    - перезаписати `self.run_dir/"findings.json"` (JSON-масив, indent=2).
    - повернути список.
  - Промпт тримати коротким (слабке залізо).

## Тести — `tests/test_hardening_agent.py` (мок LLM, без мережі)
- Фейковий LLM повертає `HardeningAdvice(mitigation="patch/upgrade", detection="ids alert")`.
- Findings: один з CVE (severity high), один без CVE (severity info).
- `test_enriches_only_prioritized`: finding з CVE отримує mitigation/detection;
  finding без CVE — лишається порожнім.
- `test_high_severity_without_cve_is_enriched`: finding severity=HIGH без CVE теж
  збагачується.
- `test_emits_selected_count`: стартова `note`-подія має `data["selected"]`, що дорівнює
  к-сті відібраних.
- `test_tolerates_llm_error`: фейковий LLM кидає `LLMError` → агент не падає, пише `error`,
  finding лишається без mitigation.
- `test_writes_findings_json`: `findings.json` існує і парситься у `list[Finding]`.
- `runs_dir=tmp_path`, `llm=<fake>`.
