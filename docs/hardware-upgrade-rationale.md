# Обґрунтування апгрейду заліза

**Дата:** 2026-07-14
**Статус:** обґрунтування рішення (на основі виміряних даних проекту)

> Коротко: поточне залізо змушує **пожертвувати якістю моделі**, усе одно **втрачає частину
> знахідок** через timeout'и, дає **повільні прогони** і **блокує пів-roadmap** (RAG,
> багатоагентну оркестрацію, більші моделі). Апгрейд знімає всі чотири обмеження.

---

## 1. Виміряні факти проекту

Джерела: [benchmarks/model-selection-recon.md](benchmarks/model-selection-recon.md) +
реальні прогони конвеєра на Metasploitable2 (`192.168.56.101`), `request_timeout=300 с`.

| Показник | 7B (`qwen2.5-coder`) | 3B (`qwen2.5`) |
|---|---|---|
| Completion (успішні LLM-виклики) | **8/23 (35%)** | 21/23 (91%) |
| Timeout'и | 15/23 (65%) | 2/23 (9%) |
| Тривалість (1 хост, 23 порти) | **~33 хв** | ~14 хв |

Реальний recon+vuln прогін на 3B: 20/23 успіху, **3 timeout'и**, ~12 хв — на **одному** хості.

**Головне:** 7B провалилась (65% timeout) **не через непридатність, а суто через залізо**.
Ми були змушені перейти на слабшу 3B, щоб система взагалі працювала.

---

## 2. Чотири збитки від поточного заліза

1. **Стеля якості аналізу.** Застрягли на 3B; глибші 7B–14B (точніший аналітичний текст,
   від якого залежить цінність звіту) недоступні.
2. **Втрата покриття.** Навіть на 3B ~9–13% findings без збагачення через timeout. У реальному
   прогоні 3 порти лишились без аналізу = **неповний звіт**, прямий ризик якості пентест-роботи.
3. **Неприйнятна пропускна здатність.** ~12–33 хв на **один** хост, суто послідовно (паралелізм
   на слабкому залізі неможливий). Екстраполяція: 10 хостів ≈ 2 год (3B) / ~5.5 год (7B).
   Довгий цикл «змінив → прогнав → побачив» гальмує і роботу, і навчання.
4. **Множення навантаження.** LLM-виклики ростуть мультиплікативно: ReconAgent — 1 виклик на
   відкритий порт (N), HardeningAgent — 1 на пріоритезований finding (P), разом ≈ N+P на хост
   (~30+ для одного хоста). Кожен новий агент і хост множать це.

## 3. Що залізо прямо блокує в roadmap

| Етап | Вимога | Чому блокується зараз |
|---|---|---|
| **M2.4 — RAG / Chroma** | embedding-модель резидентно + LLM одночасно | бракує пам'яті тримати дві моделі; ембеддинг сотень CVE-чанків повільний |
| **M4 — оркестрація (LangGraph)** | паралельні/умовні виклики агентів | немає ресурсу на конкурентність — усе строго послідовно |
| **Кореляція знахідок / глибший аналіз** | довший контекст, 7B–14B стабільно | недосяжно на поточному залізі |

Тобто ~половина запланованої цінності проекту впирається саме в hardware.

## 4. Рекомендація (під ноутбук, для локальної LLM через Ollama)

- **Apple Silicon M-series Pro/Max з ≥32 GB unified memory** — часто найкращий баланс для
  локальних LLM на ноуті: unified memory дозволяє тримати великі моделі, Metal-прискорення
  в Ollama з коробки. 48–64 GB — із запасом під RAG.
- **Або NVIDIA RTX laptop з ≥12–16 GB VRAM** (mobile 4070/4080/4090) + 32 GB RAM.
- Уникати: інтегрованої графіки / <16 GB RAM — це те, від чого тікаємо.

### Що це відкриває (на основі наших же даних)
- **Completion → ~100%:** інференс з хвилин → секунди/виклик, timeout'и зникають.
- **Повернення до 7B–14B:** якість аналізу, від якої відмовились, стає доступною без 65% провалу.
- **Пропускна здатність:** ~30 викликів/хост за 2–3 хв замість годин на 10 хостів.
- **Розблокування RAG + оркестрації** та реалістичних багатохостових сценаріїв.

### Cost/benefit
Разова інвестиція знімає **всі чотири** обмеження одночасно й розблоковує ~половину roadmap.
Альтернатива — назавжди лишитись на 3B з неповними звітами, повільними ітераціями та без
RAG/оркестрації, тобто з урізаною версією того самого проекту.

## 5. Застереження щодо чесності даних
Числа 7B/3B — з контрольованого A/B на **тому самому** скані. Оцінки для N хостів —
лінійна екстраполяція виміру на 1 хост, а не окремий вимір. Приріст від нового заліза
(секунди/виклик) — очікуваний з типової різниці інференсу для 3–14B моделей; конкретні
секунди варто підтвердити бенчмарком після апгрейду (метод готовий — completion rate з
`events.jsonl`).
