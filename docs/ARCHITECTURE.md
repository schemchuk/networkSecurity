# Архітектура проекту: AI CyberSec Lab

> Навчально-практичний проект з етичного пентесту + локальна multi-agent AI-система,
> яка аналізує реальні результати лабораторії. Кінцева ціль — портфоліо для junior/mid
> security / security+AI ролей.

**Статус:** дизайн v1
**Автор:** shemchuk
**Середовище:** Kali (основна машина) + VirtualBox (жертви) + Ollama (`qwen2.5-coder:7b`)

---

## 1. Дві траєкторії й навіщо вони разом

Проект має дві паралельні лінії, які живляться одна від одної:

1. **Hands-on pentest lab** — ти вручну проходиш ланцюг
   `recon → vuln analysis → exploitation → post-exploitation → hardening → detection`
   в ізольованій лабораторії. Кожна команда логується.
2. **Local multi-agent AI** — система локальних агентів читає *твої реальні* артефакти
   (nmap XML, логи команд) і автоматизує аналітичну роботу пентестера: парсинг,
   кореляцію з CVE/CVSS, пріоритезацію, поради з hardening/detection, генерацію звітів,
   ведення knowledge base.

**Ключовий принцип:** AI **не** замінює твій термінал і **не** генерує робочі експлойти.
Він оркеструє та аналізує. Ти лишаєшся оператором.

**Портфоліо-наратив:** «Побудував ізольовану lab з відтворюваними сценаріями + систему
локальних AI-агентів, яка перетворює сирий вивід інструментів на пріоритезований звіт із
CVE, detection- та mitigation-рекомендаціями — повністю офлайн, без хмарних API.»

## 2. Принципи дизайну (щоб проект не розповз)

1. **Спочатку вертикальний зріз, потім ширина.** Перший робочий результат — один
   наскрізний потік `nmap.xml → findings.json → report.md`, а не 5 напівготових агентів.
2. **Детерміноване окремо від імовірнісного.** Парсери (python-libnmap) дають точні
   структуровані дані. LLM лише збагачує їх. CVE-номери **завжди** з реального джерела
   (NVD/searchsploit/vulners), ніколи не «вигадані» моделлю.
3. **Append-only логи як спільна шина.** Кожен агент читає/пише події в JSONL-журнал
   прогону. Це і «пам'ять», і аудит, і те, як агенти «бачать» твою історію.
4. **Hardware-agnostic через конфіг.** Backend, модель, розмір контексту, VRAM-профіль —
   усе в `configs/`, ніде не захардкожено. Зміна моделі = зміна одного рядка.
5. **Safety by design.** Ізольована мережа, snapshot-и, системні промпти агентів
   обмежені аналізом/захистом. Межа «no weaponized payloads» вшита в політику.
6. **Кожен інструмент = навчальний артефакт.** Для кожного тула/техніки — документ за
   ланцюгом: що → коли → як читати результат → типові помилки → detection → захист.

## 3. Лабораторне середовище

```
┌───────────────────────────────────────────────────────────────┐
│  Kali (основна машина / operator)                               │
│                                                                 │
│   • термінал пентестера (nmap, metasploit, ...)                 │
│   • Ollama + qwen2.5-coder:7b   (AI-агенти тут же)              │
│   • цей репозиторій (оркестрація, парсери, KB)                  │
│                                                                 │
│   ┌─────────────── VirtualBox: ізольована мережа ────────────┐  │
│   │  vboxnet "labnet"  (Host-Only або Internal, БЕЗ NAT)     │  │
│   │                                                          │  │
│   │   [ Windows 10 victim ]   [ Metasploitable2/3 ]  ...     │  │
│   └──────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────┘
```

**Правила ізоляції:**
- Мережа жертв — **Internal Network** або **Host-Only** без доступу в інтернет.
  Жодних атак поза лабою. Це і етика, і чистота експерименту.
- Кожна жертва має **base snapshot** (чистий стан). Сценарій → прогін → відкат до snapshot.
  Відтворюваність — критична для «reproducible сценаріїв» і для тесту парсерів.
- Мішені для старту: **Metasploitable2** (швидкий, багато сервісів для recon-парсера),
  потім **Windows 10** з навмисно вразливими сервісами (для Defender/AMSI-теми).

**Логування команд (щоб агенти бачили історію):**
- `scripts/labshell` — обгортка, що запускає сесію через `script` + `asciinema`, і пише
  кожну команду в `logs/<run_id>/commands.jsonl` (timestamp, cwd, cmd, exit_code).
- `.cast`-файли asciinema лишаються для демо в портфоліо (git-ignored, окремо публікуються).

## 4. «Хребет» системи: потік даних

Усе крутиться навколо **прогону (run)** — однієї сесії роботи над однією ціллю/сценарієм.

```
  Оператор у labshell
        │  запускає nmap -oX ...
        ▼
  logs/<run_id>/raw/nmap.xml         ← сирі артефакти інструментів
        │
        ▼
  ┌──────────────┐   детерміновано
  │ tools/parsers│  (python-libnmap) → normalized host/service model
  └──────┬───────┘
         ▼
  ┌──────────────────────────────────────────────────────────┐
  │                    AGENT PIPELINE                          │
  │  Recon → VulnAnalysis → Hardening/Detection → Report       │
  │  (кожен читає/пише events.jsonl + findings.json)           │
  └──────┬─────────────────────────────────────────────┬──────┘
         ▼                                              ▼
  runs/<run_id>/findings.json                   runs/<run_id>/report.md
         │                                              │
         └──────────────┬───────────────────────────────┘
                        ▼
             knowledge_base/ (Chroma) ← KB Curator індексує
             для RAG у майбутніх прогонах
```

**Артефакти прогону** (`runs/<run_id>/`):
- `manifest.json` — run_id, час, сценарій, цілі, оператор, версії інструментів
- `raw/` — сирий вивід тулів (nmap.xml, ...)
- `events.jsonl` — append-only журнал подій усіх агентів (спільна шина/пам'ять)
- `findings.json` — нормалізовані знахідки (головна структура даних, див. §7)
- `report.md` / `report.html` — фінальний звіт

## 5. AI-система

### 5.1 Шари

```
┌─────────────────────────────────────────────────────┐
│  Orchestrator (DAG прогону агентів)                   │
├─────────────────────────────────────────────────────┤
│  Agents:  Recon │ VulnAnalysis │ Hardening │ Report   │  ← бізнес-логіка ролей
│           │ KB Curator                                │
├─────────────────────────────────────────────────────┤
│  Tools (детерміновані):                               │
│    nmap-parser │ cve-lookup │ cvss-calc │ renderer    │  ← точні операції
├─────────────────────────────────────────────────────┤
│  LLM layer:  Ollama client (qwen2.5-coder:7b)         │  ← збагачення/текст
│    + prompt templates + JSON-schema validation        │
├─────────────────────────────────────────────────────┤
│  RAG:  Chroma (vector DB) + embeddings                │  ← пам'ять/знання
└─────────────────────────────────────────────────────┘
```

### 5.2 Ролі агентів

Кожен агент = тонкий шар: (детерміновані tools) + (1 LLM-крок збагачення) + (запис у events).

| Агент | Вхід | Що робить | Вихід |
|---|---|---|---|
| **Recon** (MVP) | `nmap.xml` | парсить хости/порти/сервіси/версії; будує нормалізовану модель; LLM резюмує attack surface та пропонує *напрямки* аналізу (не експлойти) | `findings.json` (draft), events |
| **VulnAnalysis** | findings | зіставляє `сервіс+версія` → **реальні CVE** (searchsploit/vulners/NVD offline); тягне CVSS; RAG над CVE-нотатками; LLM пріоритезує та пояснює ризик | findings з CVE+пріоритетом |
| **Hardening/Detection** | findings | для кожної знахідки: mitigation + що бачить blue team (лог-артефакти, Defender/AMSI сигнали) | findings з `mitigation`, `detection` |
| **Report** | findings + manifest | збирає Markdown/HTML звіт у навчальному форматі (§10); executive summary + деталі | `report.md/html` |
| **KB Curator** | артефакти прогону + логи команд | чанкує та індексує в Chroma; будує «пам'ять» для RAG наступних прогонів | оновлений vector store |

### 5.3 Оркестрація — поетапна стратегія

- **M1–M3 (MVP):** власний тонкий orchestrator на Python — простий послідовний DAG,
  який викликає агентів і передає `findings.json`. Прозоро, легко дебажити, добре видно
  розуміння на співбесіді.
- **M4+:** міграція на **LangGraph** (граф-стейт-машина агентів) коли з'явиться розгалуження,
  повтори, умовні гілки. Не тягнути важкий фреймворк раніше, ніж він потрібен.

RAG: **Chroma** локально (простіший старт, ніж Qdrant; мігрувати за потреби). Embeddings —
локальна модель через Ollama (напр. `nomic-embed-text`), щоб лишитись повністю офлайн.

### 5.4 Робота з LLM (важливо для 7B)

- **Структурований вивід:** просимо модель повертати JSON за схемою; валідуємо через
  Pydantic; при невалідному — 1 retry з помилкою в промпті. 7B добре тримає формат при
  чіткій схемі й few-shot прикладі.
- **Розділення відповідальності:** LLM **не** визначає CVE-ID та CVSS — це з реальних даних.
  LLM отримує вже знайдені CVE і лише: пояснює, пріоритезує, пише текст.
- **Малі промпти, вузькі задачі.** Одна знахідка = один виклик, а не «проаналізуй усе».
  Це в межах контексту 7B і дає стабільнішу якість.

### 5.5 Safety guardrails

- Системний промпт кожного агента: аналіз/detection/mitigation лише; заборона генерувати
  робочий експлойт/malware/weaponized payload. При такому запиті — пояснити механізм і
  захист, без готового коду.
- Уся робота — в ізольованій lab-мережі (§3). Політика зафіксована в `configs/policy.yaml`
  і в `docs/SCOPE.md`.

## 7. Контракти даних (схеми)

Стабільні схеми — це те, що дозволяє агентам розвиватись незалежно.

### `finding` (одиниця знахідки)
```json
{
  "id": "F-0001",
  "host": "10.10.10.5",
  "port": 445,
  "protocol": "tcp",
  "service": "microsoft-ds",
  "product": "Samba",
  "version": "3.0.20",
  "source": "nmap",
  "cves": [
    { "id": "CVE-2007-2447", "cvss": 6.0, "source": "searchsploit" }
  ],
  "severity": "high",
  "priority": 1,
  "summary": "…LLM: короткий опис ризику…",
  "mitigation": "…LLM/KB: як закрити…",
  "detection": "…що бачить blue team…",
  "next_steps": ["напрямок аналізу (без готового експлойта)"],
  "evidence": "runs/<run_id>/raw/nmap.xml#port445"
}
```

### `event` (рядок у events.jsonl)
```json
{ "ts": "2026-07-09T12:00:00Z", "run_id": "…", "agent": "recon",
  "type": "finding_added|llm_call|error|note", "ref": "F-0001", "data": {} }
```

### `manifest`
```json
{ "run_id": "2026-07-09_metasploitable", "started": "…",
  "scenario": "recon-basic", "targets": ["10.10.10.5"],
  "operator": "shemchuk", "tool_versions": { "nmap": "7.9x" } }
```

## 8. Технологічний стек

| Шар | Вибір | Чому |
|---|---|---|
| LLM runtime | **Ollama** | вже стоїть, офлайн, простий API |
| Модель | **qwen2.5-coder:7b** | під твоє залізо; сильна в структурованому виводі/JSON |
| Embeddings | `nomic-embed-text` (Ollama) | офлайн, легкий |
| Vector DB | **Chroma** | простий локальний старт (Qdrant — пізніше за потреби) |
| Оркестрація | власний DAG → **LangGraph** (M4+) | не роздувати фреймворк рано |
| Парсинг nmap | **python-libnmap** | надійний детермінований парсер XML |
| CVE-джерела | **searchsploit** (exploitdb), **vulners** NSE, офлайн NVD | реальні CVE, не галюцинації |
| Валідація | **Pydantic** | схеми findings/подій |
| Звіти | Markdown → HTML (Jinja2) | просто, версіонується в git |
| Запис сесій | `script` + **asciinema** | історія команд + демо для портфоліо |

## 9. Дорожня карта (milestone-и)

- **M0 — Фундамент.** VirtualBox ізольована мережа + Metasploitable snapshot;
  `labshell` логер; `configs/model.yaml`; базовий `tools/llm.py` (тест виклику Ollama).
  *Результат:* лаба піднята, команди логуються, модель відповідає з коду.
- **M1 — MVP Recon Analyzer (вертикальний зріз).** `nmap.xml → nmap_parser → findings.json
  → report.md`. Один агент, детермінований парсер + один LLM-крок резюме.
  *Результат:* перший наскрізний артефакт для портфоліо.
- **M2 — Vuln Analysis + RAG.** `cve_lookup` з реальними CVE; Chroma KB; пріоритезація.
- **M3 — Hardening/Detection + повний Report.** Blue-team перспектива, навчальний формат звіту.
- **M4 — Orchestrator (LangGraph) + KB Curator.** Агенти в граф; індексація історії команд.
- **M5 — Портфоліо-поліш.** Case study, діаграма архітектури, санітизовані звіти-приклади,
  демо-`.cast`, README з результатами.

## 10. Навчальний формат (обов'язковий)

Для кожного інструмента/техніки — документ у `docs/learning/<tool>.md` за ланцюгом:

1. **Що це** — призначення інструмента/техніки.
2. **Коли застосовується** — на якому етапі, за яких умов.
3. **Як читати результат** — інтерпретація виводу, ключові поля.
4. **Типові помилки** — що новачки роблять не так.
5. **Detection з боку жертви** — які сліди/логи/алерти генеруються.
6. **Як захиститись** — mitigation, hardening.

Цей же формат використовує **Report Agent** для секцій звіту — навчання і продукт збігаються.

## 11. Межі та етика (див. `docs/SCOPE.md`)

- **Дозволено:** розбір механізмів, CVE-аналіз, detection, mitigation, лаб-експлуатація
  у власній ізольованій мережі.
- **Заборонено:** генерація готового working exploit/malware для реального застосування;
  будь-які дії поза lab-мережею.
- Уся діяльність — **лабораторна та освітня**.
