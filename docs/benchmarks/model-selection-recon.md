# Benchmark: вибір LLM для ReconAgent (7B vs 3B)

**Дата:** 2026-07-10
**Рішення:** робоча модель ReconAgent — `qwen2.5:3b`.

## Контекст
На слабкому залізі `qwen2.5-coder:7b` давала багато timeout'ів під час LLM-збагачення
findings (кожен відкритий порт = 1 виклик). Порівняли з меншою `qwen2.5:3b` на **тому
самому** скані (Metasploitable, 23 відкритих порти), однаковий `request_timeout=300`.

## Результати

| Метрика | 7B (`qwen2.5-coder:7b`) | 3B (`qwen2.5:3b`) |
|---|---|---|
| Run ID | `2026-07-09_131028_recon-test` | `2026-07-10_155736_recon-3b` |
| Тривалість | ~33 хв | ~14 хв |
| `llm_call` (успіх) | 8 / 23 (34.8%) | 21 / 23 (91.3%) |
| `error` (timeout) | 15 / 23 (65.2%) | 2 / 23 (8.7%) |
| Падіння (ref) | F-0001,03,04,05,06,09,10,13,14,15,16,18,19,21,22 | F-0012, F-0014 |

## Оцінка якості (осмисленість summary)
Текст 3B релевантний сервісам, не «маячня»:
- **vsftpd 2.3.4** — згадує перевірку CVE та конфігу;
- **Samba** — MS17-010 і конфігурація shares;
- **Apache 2.2.8** — відомі вразливості версії;
- **MySQL 5.0.51a** — застарілість і привілеї.

## Висновки
- **Completion rate:** 3B майже усунула timeout'и (91% проти 35%).
- **Швидкість:** ~2× швидше (менша модель).
- **Якість:** аналітичний текст адекватний для recon.
- Два падіння 3B (F-0012 `shell`/rshd, F-0014 `bindshell`) — рідкісні стендові сервіси,
  де генерація довша за timeout.

## Наслідки для конфігу
`configs/model.yaml` → `model: qwen2.5:3b`. Дизайн hardware-agnostic
(див. ARCHITECTURE.md §2) дозволив зробити це зміною одного рядка.

**Якщо потрібні останні 2 timeout'и:** підняти `request_timeout` або спробувати
`qwen2.5:1.5b` (ще швидше, ціною якості). Поточний баланс 3B — робочий.
